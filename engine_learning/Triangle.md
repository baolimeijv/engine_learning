
# 深度测试不能代替背面剔除的原因（含深度值推导）

## 1. 深度值变换推导（透视投影下的深度计算）

### 1.1 相机空间 → 裁剪空间
采用右手坐标系，相机看向 -Z，近平面距离 n > 0，远平面距离 f > 0。  
透视投影矩阵（对称视锥体）将点 `(Xe, Ye, Ze, 1)` 变换到裁剪空间：
Xc = (n / r) * Xe （r 为右平面，对称时 r = -l）
Yc = (n / t) * Ye
Zc = - (f + n) / (f - n) * Ze - (2 * f * n) / (f - n)
Wc = -Ze

### 1.2 透视除法 → NDC
x_ndc = Xc / Wc = (n / r) * Xe / (-Ze) = - (n / r) * (Xe / Ze)
y_ndc = Yc / Wc = - (n / t) * (Ye / Ze)
z_ndc = Zc / Wc = (f + n)/(f - n) + (2 * f * n)/(f - n) * (1 / Ze)

记 `A = (f + n)/(f - n)`，`B = (2 * f * n)/(f - n)`，则：
z_ndc = A + B / Ze

### 1.3 深度缓冲值（窗口深度）
OpenGL 风格深度缓冲存储 `depth = (z_ndc + 1) / 2`，代入得：
depth = (A + 1)/2 + (B/2) * (1 / Ze)
即 **`depth` 与 `1/Ze` 成线性关系**。

### 1.4 线性插值深度的正确性
在屏幕空间对 `depth` 进行重心坐标线性插值：
depth_pixel = α * depth0 + β * depth1 + γ * depth2
由于 `depth = C + D * (1/Ze)`，且 `1/Ze` 在三角形内部与重心坐标成线性关系，因此：
depth_pixel = C + D * (α*(1/Ze0) + β*(1/Ze1) + γ*(1/Ze2))
这正是 `depth` 在像素处的**精确值**。  
**结论：线性插值深度在数学上是完全正确的，不存在“透视校正”问题。**

## 2. 既然深度插值正确，为什么还会“穿透”？

- 深度测试只能比较**已生成**的深度值，不能阻止背面三角形进入光栅化阶段。
- 背面三角形与正面三角形在重叠区域的深度值**理论上**总是正面更小（更近）。  
  但由于：
  - 浮点数精度有限（`float` 约 6-7 位有效数字）
  - 三角形边缘处深度值几乎相等
  - 重心坐标插值中的累积误差
  在某些像素上，背面三角形的计算深度可能**略小于**正面三角形，从而通过深度测试，造成“穿透”。

## 3. 为什么深度测试不能剔除背面？

- **背面三角形仍会被光栅化**：即使最终被遮挡，也会消耗顶点变换、像素遍历等计算资源。
- **浮点误差导致遮挡失败**：在三角形边缘或深度值接近处，背面三角形的插值深度可能因精度问题小于正面，造成“穿透”。
- **深度测试不改变几何流程**：它只影响像素的写入，不阻止三角形进入光栅化阶段。

## 4. 实际表现（你的软渲染器案例）

- **不加背面剔除**：立方体旋转时，背面三角形被绘制，且在某些像素上通过深度测试，出现“底面颜色透到前面”。
- **加上背面剔除后**：背面三角形直接被丢弃，穿透消失，性能提升。

## 5. 类比理解

- **深度测试** = 安检门：每个人都要走过，发现有金属（深度更远）就拦下。
- **背面剔除** = 门口筛选：面朝内的直接请走，不用进门。

## 6. 结论

**深度测试不能代替背面剔除**。正确做法：先进行背面剔除（几何阶段），再对剩下的正面三角形做深度测试（光栅化阶段）。
